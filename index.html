    R — Resilience patterns (cross-cutting) Applied across every layer
    E
    Edge & traffic control ▼
    The front door - only valid, within-policy traffic gets through
    API gateway Web application firewall DDoS protection TLS termination proxy CDN Rate limiter DNS load balancer Ingress controller Bot detection
    Platform view

    Control and protection point for all external traffic. Enforces safe ingress before compute resources are consumed. Only validated, within-policy traffic reaches internal services.

    Developer view

    The first surface a developer's service is exposed through. Developers should discover and reason about traffic policy without platform team intervention.

    Platform invariant

    Only validated and within-policy traffic reaches internal services; failure defaults to blocking untrusted traffic.

    Developer invariant

    Developers can discover, test, and reason about their service's traffic policy without platform team intervention.

    C
    Compute & service architecture ▼
    Where code runs - replaceable services, controlled communication
    Container runtime Container orchestrator Service mesh Internal load balancer Sidecar proxy Service discovery registry Deployment pipeline Internal developer portal Local dev environment tool Service template / scaffold
    Platform view

    Execution environment for services. Services run stateless or state-minimised with controlled communication paths. Services are replaceable and do not rely on local state.

    Developer view

    The runtime contract a developer writes code against. A predictable, low-friction deployment target. Local-to-production parity is essential - the same mental model at every stage.

    Platform invariant

    Services are replaceable and do not rely on local state or topology assumptions.

    Developer invariant

    A developer can deploy, run locally, and debug their service using the same mental model and tooling at every stage.

    O
    Observability & diagnostics ▼
    If you can't see it, you can't fix it - every request leaves a trace
    Structured log aggregator Metrics collection agent Time-series database Distributed tracing system Alerting engine Dashboarding platform On-call routing tool Error tracking service Telemetry SDK
    Platform view

    Actionable visibility into system behaviour and failure modes. Every request carries traceable context across all service boundaries.

    Developer view

    The primary tool a developer has for understanding whether their service is working and why it failed. Production must be legible without understanding the telemetry pipeline.

    Platform invariant

    Every request carries traceable context across all service boundaries.

    Developer invariant

    A developer can diagnose a production failure using platform-provided tooling without needing platform team assistance.

    D
    Data & storage ▼
    State with a contract - know your consistency guarantees before you commit
    Relational database Document store Key-value store Wide-column store Distributed cache Object / blob storage Search index Schema migration tool Connection pooler Point-in-time recovery
    Platform view

    Persistent and cached state management within defined consistency and durability bounds. Every dataset has explicit and documented consistency and failure behaviour under partition.

    Developer view

    The place where application behaviour is shaped by constraints a developer may not have chosen themselves. Developers must know what guarantees they are getting before committing to a storage choice.

    Platform invariant

    Every dataset has explicit and documented consistency and failure behaviour under partition.

    Developer invariant

    Developers know exactly what guarantees they are getting and giving up before committing to a storage choice.

    E
    Eventing & async processing ▼
    Decouple to survive - consumers must handle duplicates and delays gracefully
    Message queue Service bus Distributed log / event stream Dead-letter queue Event schema registry Background job processor Cron / scheduled task runner Consumer lag monitor Replay tooling
    Platform view

    Decoupled communication through asynchronous messaging and streams. Enables resilient workflows without synchronous dependency chains. All consumers must tolerate duplicate and delayed messages safely.

    Developer view

    A coordination mechanism that moves complexity out of individual services but introduces its own operational surface. Async failure modes must be legible and testable without deep expertise in the messaging platform.

    Platform invariant

    All consumers must tolerate duplicate and delayed messages safely.

    Developer invariant

    Developers can inspect, replay, and diagnose their consumers without requiring platform team involvement.

    D
    Deployment - control plane & configuration ▼
    Intent becomes reality - configuration is code, and drift is a bug
    GitOps controller Infrastructure-as-code tool Secret management system Configuration store Feature flag service Policy engine Drift detection tool Change preview / dry-run Audit log store
    Platform view

    System-wide configuration, orchestration, and state convergence. Desired system state is consistently applied and auditable. Drift is a failure condition, not an accepted state.

    Developer view

    The mechanism through which intent becomes running system behaviour - and the source of unexpected surprises when it drifts. Configuration changes must be discoverable and safe without platform team gatekeeping.

    Platform invariant

    System state converges deterministically toward declared configuration with full auditability.

    Developer invariant

    Developers can propose, preview, and understand the effect of a configuration change before it reaches production.

    Resilience - applied across every layer above ▼
    Circuit breaker library Retry middleware Bulkhead / thread pool isolation Timeout policy engine Chaos / fault injection framework Load shedding proxy Health check endpoint standard Graceful degradation toggle Backoff strategy
    Platform view

    System behaviour under partial or total failure conditions. Prevent cascading failures and preserve system usability under stress. No single dependency failure is allowed to exhaust system-wide resources.

    Developer view

    The difference between a developer's service degrading gracefully and taking down everything that depends on it. Platform-provided resilience defaults must be safe out of the box - opt out, never opt in.

    Platform invariant: no single dependency failure may exhaust system-wide resources
    Developer invariant: resilience defaults are safe out of the box - opt out, never opt in
